The current Enterprise product is strongest around **knowledge operations**.

At a high level, the flow is:

```text
source or uploaded file
  -> import or sync job
  -> normalized document + chunks
  -> KB page and provenance links
  -> optional RAGFlow index
  -> /kb reader, /v1/kb APIs, and /mcp tools
```

## Practical First Workflow

<Steps>
  1. Sign in to `/admin`.

  2. Open `知识运营`.

  3. Import a document or prepare a connector-backed source.

  4. Watch the import or sync job status until it finishes.

  5. Open `/kb` to confirm the imported content is readable in the browser.

  6. If your deployment uses RAGFlow, verify the dataset and retrieval path from the admin knowledge views before treating search quality as production-ready.
</Steps>

## Good Local Seed Content

The `haloforge-docs` repository is a practical early knowledge seed because it already contains structured Markdown/MDX pages, sections, and navigation. It is a good candidate for a Starlight-backed knowledge source while Enterprise KB workflows are still being shaped.

## What `/kb` Is For

The KB reader is not the admin console. It is a separate authenticated surface for:

- project trees
- page reading
- search
- related pages
- backlinks
- citations
- provenance

That means a healthy Enterprise deployment already has two browser surfaces:

- `/admin` for operators
- `/kb` for readers and knowledge validation

## When MCP Matters

Use `/mcp` when the consumer is an AI client, tool runner, or automation rather than a human browsing the site. MCP is the supported path for `kb.search`, project listing, page reading, and citation-aware retrieval in tool-driven clients.
